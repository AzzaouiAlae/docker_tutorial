1. Things that "work" while breaking the model

Every pattern on this page produces a container that runs, serves traffic, and looks fine in docker ps. That's exactly what makes them dangerous to learn from trial and error: nothing crashes immediately, so there's no obvious signal that you've sidestepped the isolation, networking, or lifecycle guarantees Docker is supposed to give you.

A working container and a correct container aren't the same thing. Each pattern below trades away something Docker exists to provide — isolation, service discovery, or a meaningful running/stopped state — usually to dodge a problem that has a real fix elsewhere.

2. network: host as a shortcut

# Looks like it solves "the container can't reach my service" instantly:
services:
  app:
    network_mode: host

host networking makes the container share the host's network namespace directly — no virtual interface, no isolation, no port mapping. It's a real driver with real uses (Tutorial 5, section 2) for performance-sensitive single-purpose tools. Reached for as a fix for "my multi-container app can't talk to itself," it's an anti-pattern: it papers over a user-defined bridge network and DNS-based service discovery that were never set up correctly, and it silently removes the isolation between every container using it and the host's own network stack.

If two services genuinely can't reach each other, the fix is almost always a shared user-defined bridge network and the right service name — not collapsing the network namespace boundary entirely.

3. --link instead of a user-defined network

# Legacy flag, still works, shouldn't be reached for:
docker run --link mariadb:db wordpress

--link predates user-defined bridge networks and Docker's embedded DNS server. It wires up a one-directional connection and a hardcoded /etc/hosts entry on the linking container — fragile, undocumented in current Docker releases, and replaced entirely by just attaching both containers to the same user-defined network (Tutorial 5, section 3), which gives every container on it two-way name resolution automatically.

There's no scenario where --link does something a shared network can't do better. Seeing it in a Dockerfile or run command is a sign the network setup was never actually fixed — just worked around.

4. Infinite-loop entrypoints

# All three keep the container "running" without it doing anything:
CMD ["tail", "-f", "/dev/null"]
CMD ["sleep", "infinity"]
CMD ["sh", "-c", "while true; do sleep 1; done"]

Each of these is reached for the same way: something needs to start first, or the real command keeps exiting, so a no-op loop becomes PID 1 to keep docker ps showing "Up". This directly contradicts the PID 1 model from Tutorial 3, section 4 — PID 1 existing is not the same as the service being healthy or even present. A container "running" a busy-loop has no actual workload inside it; docker exec into it and there's nothing listening on any port.

The usual root cause is either: (1) the entrypoint script never reaches its final exec into the real service because setup is failing silently, or (2) someone used the container as a long-lived debugging shell instead of fixing the actual command that should be PID 1. Both are worth tracking down directly rather than papering over with a loop.

5. Why these specifically are worth calling out

All three share the same shape: a real, sometimes-legitimate Docker feature (host networking, a legacy flag, a shell loop) used to avoid fixing a different problem (network configuration, DNS setup, a broken entrypoint) rather than for its actual purpose. None of them throw an error. All of them pass a casual "does it run?" check. That combination — silent, plausible-looking, and wrong — is what makes them worth specifically watching for in review, not just something a test suite will catch.

6. Spotting them

# Check what network mode a container is actually using
docker inspect -f '{{.HostConfig.NetworkMode}}' some_container

# --link shows up here if anything still uses it
docker inspect -f '{{json .HostConfig.Links}}' some_container

# A container "running" a busy-loop has nothing meaningful in its process list
docker top some_container
