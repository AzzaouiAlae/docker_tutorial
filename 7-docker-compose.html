1. What Compose solves

Without Compose, running several interconnected services by hand means a separate docker build and docker run command per service, each with matching network, volume, env, and secret flags, executed in the right order, every single time. Compose lets you declare that entire multi-container setup once, in docker-compose.yml, and bring it all up or down with one command.

2. A standalone container vs. one managed by Compose

docker run mariadb and a mariadb: service block in docker-compose.yml both end up starting the same image as a container — but they aren't equivalent in practice, because Compose changes several defaults around that container:

Aspectdocker run (standalone)Compose-managed
NetworkingDefault bridge network unless you pass --network — no automatic DNS by nameAttached to the project's user-defined network automatically, with DNS resolution by service name (Tutorial 5)
NamingRandom name unless you pass --namecontainer_name:, or a predictable project_service_1 default
LifecycleYou track and restart it yourself; --restart set manuallyrestart: policy declared once, reapplied consistently every up
DependenciesYou sequence the commands yourselfdepends_on orders starts/stops across the whole stack
Reproducing the setupRe-type (or re-script) every flag by handdocker compose up reads the same flags back out of the file
A single docker run is still the right tool for a one-off, throwaway container (testing an image, debugging in isolation). The moment a container needs to consistently talk to siblings, restart the same way every time, and come up together with them, that's the signal to put it in docker-compose.yml instead of a long-lived hand-typed command.

3. Anatomy of docker-compose.yml

A compose file has up to four top-level blocks:

services:    # one entry per container — the bulk of the file
  nginx:
    ...

volumes:     # named volumes referenced by services (Tutorial 4)
  wordpress_data:
    ...

networks:    # named networks referenced by services (Tutorial 5)
  app_network:
    ...

secrets:     # files mounted read-only into services (Tutorial 9)
  db_password:
    ...

4. Common service keys

5. depends_on and start order

depends_on controls only the order containers are started in — by default it does not wait for the dependency's application to be ready (covered in depth in Tutorial 5, section 6). Compose still starts dependencies first and stops dependents first, which is enough to express "nginx's container shouldn't even attempt to start until wordpress's container exists." 

nginx:
  depends_on:
    - wordpress
    - resume
    - grafana
    - adminer

6. A real service block, explained

A typical nginx service in front of a WordPress stack looks like this:

nginx:
  build: ./nginx
  image: nginx:1.0
  container_name: nginx
  restart: on-failure
  init: true
  secrets:
    - nginx_crt
    - nginx_key
  depends_on:
    - wordpress
  ports:
    - "${NGINX_PORT}:${NGINX_PORT}"
  volumes:
    - wordpress_data:/var/www/
  networks:
    - app_network
  env_file:
    - .env
  • build + image — nginx is built from your own Dockerfile, not pulled, and tagged nginx:1.0.
  • secrets: nginx_crt / nginx_key — the TLS certificate and key are injected as files, not baked into the image or passed as env vars (Tutorial 9).
  • depends_on — nginx proxies to wordpress, so that container must at least have started first.
  • ports: "${NGINX_PORT}:${NGINX_PORT}" — the only port exposed to the host; everything else stays internal to app_network.
  • volumes: wordpress_data:/var/www/ — nginx reads WordPress's PHP files directly off the same volume WordPress writes to, so it can serve static assets without proxying every request.
  • env_file: .env — pulls in the variables covered in Tutorial 8, including ${NGINX_PORT} used right above it.

Compare with a mariadb service that has no ports: key at all — it's reachable only from other containers on app_network, never from outside the host, which is good security practice for a database.

7. Useful commands

# Build (or rebuild) all images and start every service
docker compose up --build

# Run detached (in the background)
docker compose up -d

# Stop and remove containers, networks (keeps volumes)
docker compose down

# Stop and remove containers, networks, AND named volumes — destroys data
docker compose down -v

# Logs for one service, or all of them
docker compose logs -f wordpress
docker compose logs -f

# Rebuild just one service's image
docker compose build mariadb

# Validate the file without running anything
docker compose config

8. Wrapping Compose in a Makefile

docker compose commands are long and easy to fat-finger, so it's common to wrap the handful you actually use day to day behind short make targets: 

COMPOSE = docker compose -f srcs/docker-compose.yml

build:
	$(COMPOSE) build

up:
	$(COMPOSE) up -d

down:
	$(COMPOSE) down

clean: down
	docker system prune -f

fclean: clean
	$(COMPOSE) down -v
	docker system prune -af --volumes

re: fclean up

.PHONY: build up down clean fclean re
  • up / down — the two you run constantly; matching short names to the Compose subcommands they wrap.
  • clean — stops the stack and prunes dangling images/build cache, without touching volumes.
  • fclean — the destructive one: down -v plus a full prune, deleting volumes too. Named so it reads as strictly worse than clean, not a synonym for it.
  • re — the everyday "start over" target: tear everything down (including volumes) and bring it back up fresh.
Naming consistency matters more than the specific names. Whatever convention you pick, keep down -v (volume-destroying) behind a target that's clearly more destructive than your plain down/clean — a teammate (or future you) running make clean expecting to keep their data is exactly the kind of mistake this is meant to prevent.
