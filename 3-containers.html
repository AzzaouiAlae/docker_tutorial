1. What a container actually is

A container is an image plus one writable layer, running as an isolated process tree on the host kernel. Nothing about a container is virtualized hardware — it's dockerd calling Linux kernel APIs (namespaces, cgroups, sometimes seccomp/AppArmor) around a normal process. Stop the container and that writable layer's changes are gone unless you wrote them to a volume (Tutorial 4) or bind mount.

Containers are ephemeral by design. Treat the container's own filesystem as disposable. Anything that must survive a restart — a database's data directory, WordPress's uploads — belongs in a volume, not in the container's writable layer.

2. Container lifecycle

docker create ──► Created │ docker start ▼ Running ──► docker pause ──► Paused │ │ docker unpause │ docker stop / process exits ▼ ▼ Running Exited (stopped, layer kept) │ docker rm ▼ (gone)

docker run is really create + start in one call. A stopped container still exists on disk (you can inspect its logs, restart it) until you docker rm it.

4. PID 1 and why your service must stay foreground

The process named in CMD/ENTRYPOINT becomes PID 1 inside the container. When that process exits, the container exits — Docker doesn't keep a container alive just because something is technically still running inside it. This is exactly why "no tail -f, no infinite sleep loop" is a hard rule in good Dockerfile practice: PID 1 must be your actual service, running in the foreground. 

# Wrong — container exits as soon as bash returns, nginx never really "owns" the container
CMD ["sh", "-c", "nginx && sleep infinity"]

# Right — nginx itself is PID 1, runs in the foreground
CMD ["nginx", "-g", "daemon off;"]
PID 1 also has a special responsibility: reaping zombie processes and forwarding signals (SIGTERM on docker stop). Services not designed to be PID 1 sometimes need a flag (daemon off; for nginx, --nodaemonize for some services) so they handle signals correctly instead of orphaning children.

5. Restart policies

PolicyBehavior
noDefault — never restart automatically
on-failureRestart only if the process exits with a non-zero code
alwaysAlways restart, including after a Docker daemon restart
unless-stoppedLike always, but respects a manual docker stop
on-failure is often the safer default in a Compose stack: narrower than always on purpose. A container that exits 0 (a deliberate stop) stays stopped, while a crash (non-zero exit) gets retried. A misconfigured service then doesn't mask itself behind an infinite restart loop that looks like "it's running" in docker ps.

6. Getting inside a running container

# Run an interactive shell inside an already-running container
docker exec -it mariadb bash

# View logs (essential for debugging crash loops)
docker logs wordpress
docker logs -f wordpress    # follow, like tail -f

# Copy files in/out without exec
docker cp wordpress:/var/www/html/wp-config.php .

7. Bootstrapping an app on first start

A container image is reused to start the same service over and over, but some setup — creating a database and its users, writing a first-run config file, seeding an admin account — should only happen once, the very first time that service's data directory is empty. Doing this safely is the job of an entrypoint script that runs as PID 1: it checks whether setup has already happened, does it if not, and then execs into the real foreground process so the service itself ends up as PID 1, instead of leaving a setup script running with the service as its background child. 

#!/bin/sh
set -e

DB_DIR=/var/lib/mysql

if [ ! -d "$DB_DIR/mysql" ]; then
  # First run: the data directory is empty, so nothing has been initialized yet.
  mariadb-install-db --datadir="$DB_DIR" --user=mysql

  # Start mariadbd briefly, in the background, just to run setup SQL against it.
  mariadbd --datadir="$DB_DIR" --user=mysql --skip-networking &
  pid="$!"
  until mariadb-admin ping --silent 2>/dev/null; do sleep 0.5; done

  mariadb <<-SQL
    CREATE DATABASE IF NOT EXISTS app;
    CREATE USER IF NOT EXISTS 'app_user'@'%' IDENTIFIED BY '$(cat /run/secrets/db_password)';
    GRANT ALL ON app.* TO 'app_user'@'%';
SQL

  # Stop the temporary instance; the real one below takes over as PID 1.
  mariadb-admin shutdown
  wait "$pid"
fi

# exec replaces the shell — mariadbd becomes PID 1 for the life of the container
exec mariadbd --datadir="$DB_DIR" --user=mysql
  • The guard (if [ ! -d ... ]) — checking for evidence that setup already ran is what makes the script idempotent. Without it, every container restart would try to recreate the database and fail, or silently reset state.
  • A throwaway instance for setup SQL — the database needs to actually be running to execute CREATE USER/GRANT, so the script starts it temporarily, runs the SQL, then shuts it down before handing off to the real process.
  • exec at the end — this is the same exec-form distinction from Tutorial 6: without it, the script stays PID 1 and the real service runs as its child, which breaks signal handling on docker stop.
This exact shape — guard, one-time setup, exec into the foreground process — is what's behind most official images' entrypoints (MariaDB's, Postgres's, WordPress's). Writing your own version of it is really just making that pattern explicit instead of relying on an upstream image to do it for you.

8. Useful commands

# List running / all containers
docker ps
docker ps -a

# Stop, start, restart, remove
docker stop wordpress
docker start wordpress
docker restart wordpress
docker rm wordpress

# Resource usage live
docker stats

# Process list inside a container's PID namespace
docker top wordpress
