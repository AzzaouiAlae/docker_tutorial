1. What a Dockerfile is

A Dockerfile is a plain-text script of instructions that docker build executes in order to produce an image. Each instruction normally creates one new layer on top of the previous state (Tutorial 2). Writing your own Dockerfile, rather than pulling a finished image of the service itself, is what makes the build process inspectable and reproducible.

2. Core instructions

InstructionPurpose
FROMThe base image everything else builds on. Must be the first real instruction.
RUNExecutes a command at build time and commits the filesystem change as a layer.
COPYCopies files from the build context (your host) into the image.
ADDLike COPY, but can also fetch URLs and auto-extract archives — prefer COPY unless you specifically need that.
WORKDIRSets the working directory for subsequent instructions and the container's default CWD.
EXPOSEDocuments which port the container listens on (doesn't publish it — that's Compose's ports:).
ENVSets an environment variable baked into the image. Covered further in Tutorial 8.
USERSwitches which user subsequent instructions and the container's process run as.
CMD / ENTRYPOINTWhat runs when the container starts. See section 4.

3. Layer caching and instruction order

Docker caches each layer and reuses it on the next build if the instruction and its inputs are unchanged. As soon as one instruction's cache misses (e.g. a changed source file), every instruction after it also misses, even if they wouldn't have changed. So order matters:

Good order (rarely-changing → frequently-changing): FROM alpine:3.23 RUN apk update ← cached almost always RUN apk add --no-cache mariadb ... ← cached almost always COPY ./tools/script.sh /app/ ← only this re-runs when you edit the script CMD ["/app/script.sh"] Bad order — any source edit invalidates the expensive apk install too: FROM alpine:3.23 COPY . /app RUN apk update RUN apk add --no-cache mariadb ...
The cache hit is blind to the outside world. Docker decides "reuse this layer" purely from the instruction text and the parent layer — it has no idea Alpine's actual package index changed since your last build. A long-lived cache hit on RUN apk update can quietly mean you're building against a months-old package snapshot. If you need guaranteed-fresh packages, use docker build --no-cache periodically rather than trusting the cache forever.

4. CMD vs. ENTRYPOINT

Both define the container's startup process, but they compose differently:

  • CMD alone — the default command, fully overridable by anything passed after the image name in docker run.
  • ENTRYPOINT alone — always runs; anything from docker run is appended as arguments to it.
  • Both together — ENTRYPOINT is the fixed program, CMD supplies its default arguments (overridable).
# Exec form (preferred): no shell wrapper, the binary becomes PID 1 directly
CMD ["/app/script.sh"]

# Shell form: runs via "/bin/sh -c", which becomes PID 1 instead of your process
CMD /app/script.sh
The exec form matters for the PID 1 / signal-handling reasons covered in Tutorial 3 — a shell wrapper as PID 1 can swallow or mishandle SIGTERM on docker stop.

5. A mariadb Dockerfile, line by line

FROM alpine:3.23

RUN apk update
RUN apk add --no-cache mariadb mariadb-client
RUN mkdir -p /run/mysqld
RUN chown -R mysql:mysql /run/mysqld

COPY ./tools/script.sh /app/script.sh

RUN chmod +x /app/script.sh

CMD ["/app/script.sh"]
  • FROM alpine:3.23 — a minimal base, keeping image size and attack surface small.
  • apk update / apk add — installs MariaDB server and client from Alpine's own package repo, not a pre-built MariaDB image.
  • mkdir + chown /run/mysqld — MariaDB needs this socket directory to exist and be owned by the mysql user before it starts, or it refuses to launch.
  • COPY script.sh + chmod +x — brings in your own entrypoint logic (first-run DB initialization, user/permissions setup) instead of relying on an upstream image's init scripts.
  • CMD ["/app/script.sh"] — exec form, so your script (which should itself exec mariadbd ... at the end) ends up as PID 1.
Notice there's no EXPOSE 3306 here — that's fine; EXPOSE is purely documentation. What actually matters for reachability is the network attachment from Tutorial 5, not this Dockerfile.

6. Practices that actually matter

  • No :latest — pin a specific base image version (alpine:3.23, not alpine), so your build is reproducible months from now.
  • One process per Dockerfile/container — don't install nginx and PHP in the same image meant to also run MariaDB.
  • End with a foreground process — no tail -f /dev/null, no sleep infinity as the final command.
  • Minimize layers sensibly — combining trivially related RUN commands with && reduces layer count, but don't sacrifice readability or caching granularity to chase a smaller layer count.

7. Useful commands

# Build with a tag, from a given build context directory
docker build -t mariadb:1.0 ./mariadb

# Rebuild ignoring the cache entirely (useful when debugging a stale layer)
docker build --no-cache -t mariadb:1.0 ./mariadb

# See exactly which instruction produced which layer, and its size
docker history mariadb:1.0
