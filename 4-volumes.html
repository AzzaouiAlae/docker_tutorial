1. Why containers alone aren't enough for data

A container's writable layer lives and dies with that container. docker rm a MariaDB container and its /var/lib/mysql data — every WordPress post, every user — disappears with it. Volumes solve this by moving the data outside the container's lifecycle, into storage managed by the Docker host that survives container removal and rebuilds.

This is the whole reason volumes exist: any data that must survive a docker compose down / up cycle, a crash, or a rebuild has to live in a volume, not in the container's own writable layer.

2. Named volumes vs. bind mounts vs. tmpfs

TypeWhere data livesManaged byTypical use
Named volume/var/lib/docker/volumes/...DockerDatabase files, persistent app state
Bind mountAny host path you chooseYouSharing host config/code, e.g. /srv/app/data
tmpfs mountHost RAM, never written to diskDockerSecrets/temp data that must never persist
Some setups require volumes mounted at a specific, known host path rather than wherever Docker decides to put them. That's still a bind mount with a fixed location, even though Compose's volumes: syntax for it (a named entry with driver_opts) looks similar to a plain named volume.

3. Named volumes in depth

A named volume is a directory Docker manages for you, identified by name instead of path. Docker decides where it physically lives; you just reference it by name from any container. 

docker volume create mariadb_data
docker run -v mariadb_data:/var/lib/mysql mariadb

Multiple containers can mount the same named volume simultaneously — useful for sharing data between services without a bind mount tied to one specific host path.

4. Bind mounts in depth

A bind mount maps an exact host directory into the container. Changes are visible instantly on both sides — it's literally the same files, not a copy. 

docker run -v /srv/app/data/wordpress:/var/www/html wordpress

# Equivalent, more explicit syntax
docker run --mount type=bind,source=/srv/app/data/wordpress,target=/var/www/html wordpress
Permissions gotcha: bind-mounted directories keep the host's ownership and permissions. If the host directory is owned by your user but the process inside the container runs as a different UID (e.g. www-data), you can get permission-denied errors even though the path is correctly mounted.

5. Volumes in docker-compose.yml

services:
  mariadb:
    volumes:
      - mariadb_data:/var/lib/mysql

volumes:
  mariadb_data:
    driver: local
    driver_opts:
      type: none
      o: bind
      device: /srv/app/data/mariadb
This pattern — a top-level volumes: entry whose driver_opts point at a fixed host path — gives you a Compose-managed volume name and a guaranteed, inspectable host location. Useful whenever you need to know exactly where on disk a service's data lives, rather than letting Docker pick the path for you.

6. Bind-mounting a fixed host path

A database's data directory or an app's web root is often backed by a volume defined this way, so that:

  • docker compose down (without -v) removes containers but leaves the volumes and their data untouched.
  • A fresh docker compose up reattaches the same volumes, and the app/database pick up right where they left off instead of re-running first-time setup.
  • The data is physically sitting under a known path on the host, inspectable and backup-able outside of Docker entirely.
Destructive command to know about: docker compose down -v deletes the named volumes too — that's the one command that actually wipes this kind of persisted data. Never run it casually on a stack you care about.

7. Proving the data actually survives

It's easy to believe a volume persists data without ever checking. Recreating the container and confirming the data is still there only takes a minute, and it's the only way to actually know your setup is correct rather than assumed correct: 

# 1. Write something identifiable through the running container
docker exec mariadb sh -c "echo hello > /var/lib/mysql/proof.txt"

# 2. Remove the container entirely — NOT the volume
docker compose rm -sf mariadb

# 3. Recreate it fresh from the same compose file
docker compose up -d mariadb

# 4. Confirm the file is still there, in the new container
docker exec mariadb cat /var/lib/mysql/proof.txt   # → hello

The new container in step 3 has a brand-new writable layer — none of its own filesystem history carries over. The only reason proof.txt survives is that it was never actually stored in that writable layer; it was in the named volume, which step 2 left untouched.

If you instead ran docker compose down -v between steps 1 and 3, the file would be gone — that's the volume itself being deleted, not just the container. Re-running this same check after a down -v is a quick way to see the difference firsthand.

8. Useful commands

# List, inspect, remove volumes
docker volume ls
docker volume inspect mariadb_data
docker volume rm mariadb_data

# Remove all unused (unreferenced) volumes
docker volume prune

# See exactly what's mounted where on a running container
docker inspect -f '{{json .Mounts}}' mariadb
