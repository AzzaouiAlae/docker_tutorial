1. What is a Docker network?

Every container gets its own network namespace — its own loopback interface, routing table, and IP address — isolated from the host and from other containers by default. A Docker network is a virtual switch that connects the namespaces of selected containers together so they can talk to each other, while staying isolated from containers that aren't on that network.

Host machine ├── eth0 (physical/host NIC) └── docker0 / br-xxxx (virtual bridge = the "network") ├── veth → container A (nginx) 10.0.5.2 ├── veth → container B (wordpress) 10.0.5.3 └── veth → container C (mariadb) 10.0.5.4

2. Network drivers

DriverUse case
bridgeDefault for standalone/Compose containers on a single host. What you're using.
hostContainer shares the host's network namespace directly — no isolation, no port mapping needed.
noneNo networking at all — fully isolated container.
overlayConnects containers across multiple Docker hosts (Swarm mode).
macvlanGives a container its own MAC address on the physical LAN.
In practice: a single-host bridge network is the right default for most Compose stacks — all your services run on one machine, and you just need them to find each other.

3. User-defined bridge networks

Docker has a default bridge network, but it's intentionally limited: containers on it can only reach each other by IP, not by name. A user-defined bridge (what app_network is) fixes that and adds:

  • Automatic DNS resolution by container/service name
  • Better isolation — only containers explicitly attached can communicate
  • Ability to attach/detach containers without restarting them
# Equivalent manual commands to what Compose does for you:
docker network create app_network
docker run --network app_network --name mariadb mariadb
docker run --network app_network --name wordpress wordpress

4. DNS-based service discovery

On a user-defined bridge, Docker runs an embedded DNS server (127.0.0.11 inside each container). Any container on the same network can resolve another container's name (or Compose service name) straight to its internal IP — no hardcoded IPs, no /etc/hosts editing. 

# Inside the wordpress container, this just works:
ping mariadb
mysql -h mariadb -u wp_user -p
This is exactly why your WordPress config can use DB_HOST=mariadb instead of an IP — Compose registers each service's container name as a DNS entry on app_network.

5. Networks in docker-compose.yml

Declaring a network in Compose has two parts: a top-level definition, and per-service attachment.

networks:
  app_network:
    name: app_network   # fixed name instead of "srcs_app_network"

services:
  nginx:
    networks:
      - app_network
  wordpress:
    networks:
      - app_network
Without a top-level networks: block, Compose still creates a default network automatically (named <project>_default) and attaches every service to it. Defining app_network explicitly just gives you a predictable, fixed name and lets you control its driver/options.

6. depends_on vs. network readiness

depends_on only controls start order (and with condition: service_healthy, readiness) — it does not wait for the application inside the container to be ready to accept connections. 

wordpress:
  depends_on:
    - mariadb   # mariadb's container starts first, but MySQL itself
                # might not be ready to accept connections yet
Common pitfall: WordPress trying to connect to MariaDB before MySQL has finished initializing. The network connection itself isn't the problem — name resolution works immediately — it's that nothing is listening on port 3306 yet. Fix with a healthcheck + condition: service_healthy, or retry logic in your entrypoint script.

7. Useful commands

# List all networks
docker network ls

# Inspect a network: connected containers, subnet, gateway
docker network inspect app_network

# See which networks a running container is on
docker inspect -f '{{json .NetworkSettings.Networks}}' nginx

# Test name resolution / connectivity from inside a container
docker exec -it wordpress ping -c1 mariadb
docker exec -it wordpress nslookup mariadb

# Manually attach a running container to a network
docker network connect app_network some_container

8. Putting it together

In a typical docker-compose.yml, every service lists:

networks:
  - app_network

and the network itself is declared once at the bottom:

networks:
  app_network:
    name: app_network

What this gives you, for a stack like nginx + wordpress + mariadb + redis:

  • All services share one isolated bridge network.
  • Each container reaches the others by service/container name — e.g. nginx does FastCGI to wordpress:9000 (php-fpm), and wordpress connects to mariadb:3306 and redis:6379.
  • Only the ports you explicitly publish (ports:) are reachable from outside the host — everything else stays internal to app_network, which is good security practice (e.g. a database with no published port).
Why one shared network instead of several? A single network is usually enough so every service can reach every other service it needs, without managing multiple network definitions. You'd only split into separate networks if you actually want to isolate groups of services from each other (e.g. keeping a monitoring stack off the same network as a public-facing one).
