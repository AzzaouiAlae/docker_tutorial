1. The problem before containers

Say you need to run a web server, a database, and an application server together — all on one machine, all talking to each other. Before containers, you'd install every one of those directly on the host: different package versions, different config file locations, different init systems, and a real chance that installing one service's dependency breaks another's. "Works on my machine" became a running joke for exactly this reason.

The core pain: software needs a very specific environment (OS libraries, runtime versions, config) to run correctly, and that environment is hard to reproduce identically on another machine — or even on the same machine six months later.

2. What Docker actually is

Docker is a tool for packaging an application together with everything it needs to run — code, runtime, system libraries, config — into a single unit called an image, and then running that unit as an isolated process called a container.

The key idea: a container is not a separate computer. It's a normal process on your host's Linux kernel, just fenced off using kernel features (namespaces, cgroups) so it gets its own filesystem, network interface, and process tree — while still sharing the host's kernel directly, with none of a VM's overhead.

3. Docker vs. a virtual machine

Virtual MachineDocker container
What it virtualizesEntire hardware + its own kernelJust the process, sharing the host kernel
Boot timeSeconds to minutesMilliseconds to seconds
SizeGigabytes (full OS image)Megabytes (just app + deps)
Isolation strengthStrong — separate kernelWeaker — shared kernel, namespaced
Typical useOne heavyweight isolation boundary (a whole guest OS)Many lightweight processes sharing one kernel
In practice: a VM is often the outer boundary (a cloud instance, a lab machine), and Docker gives you lightweight isolation inside that VM for each service, instead of installing a web server, a database, and an app all loose on one filesystem.

4. The moving pieces: image, container, engine

  • Image — a read-only template: filesystem snapshot + metadata (what command to run, what ports to expose). Built once, reused everywhere. Covered in Tutorial 2.
  • Container — a running (or stopped) instance of an image, with its own writable layer on top. Covered in Tutorial 3.
  • Docker Engine — the daemon (dockerd) running on your host/VM that builds images and runs containers, talked to via the docker CLI.
# The engine is what you're actually talking to:
docker version
docker info
Mental shortcut: an image is the recipe, a container is the dish you cooked from it. You can cook the same recipe (image) into many independent dishes (containers) — that's exactly what lets you run several instances of the same service from one image.

5. Why Docker, in practice

  • One service per container is the convention that makes Docker's unit of work — build, scale, restart, replace — match a single responsibility, instead of one fat container doing five jobs.
  • Reproducibility — a Dockerfile is the single source of truth for how a service is built and configured. Anyone gets the exact same setup by running docker compose up, instead of following a setup doc that drifts out of date.
  • Isolation without VM overhead — many services run side by side on one host, each with its own filesystem and process space, without paying for a separate VM per service.
  • Composable networking and storage — instead of services finding each other via hardcoded IPs and shared host directories, Docker gives you named networks and volumes (Tutorials 4 and 5) as first-class, declarative building blocks.
  • It scales the same mental model from one container to a whole stack — the same image/container/network/volume primitives apply whether you're running one service locally or a dozen in production.
What Docker is not for: it's not a security sandbox against malicious code, and it's not a substitute for understanding the services you're containerizing. You still need to know why a database needs persistent storage or why a PHP app needs PHP-FPM — Docker just gives you a clean, repeatable place to express that knowledge.

6. First commands to try

# Confirm the engine is alive and see resource usage
docker info
docker version

# Pull and run something trivial to see a container exist and exit
docker run hello-world

# See what's currently running on your machine
docker ps
docker ps -a   # include stopped containers
