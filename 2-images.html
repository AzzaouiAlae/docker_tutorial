1. What an image actually is

An image is a read-only, immutable template for a filesystem plus metadata: the default command to run, exposed ports, environment defaults, the working directory, the user to run as. It contains no running process — it's a blueprint, not a thing in motion. When you docker run an image, Docker takes that template and adds one writable layer on top: that combination is a container (Tutorial 3).

Every Dockerfile you write is the recipe for exactly one image. In a multi-service stack (a web server, an app, a database), each service gets its own Dockerfile and its own image — that's what keeps them independently buildable, taggable, and replaceable.

2. Layers and the union filesystem

An image isn't one big blob — it's a stack of read-only layers, each one a diff from the layer below it. Every instruction in a Dockerfile that changes the filesystem (RUN, COPY, ADD) produces a new layer. Docker stitches them together with a union filesystem so the container sees one consistent view.

Image "myapp:1.0" ├── layer 4: COPY conf.d/www.conf ← your customization ├── layer 3: RUN apt install php-fpm ← your customization ├── layer 2: RUN apt update ← your customization └── layer 1: debian:bullseye-slim base ← base image, downloaded once
Why this matters: layers are cached and shared. If three of your Dockerfiles start FROM debian:bullseye, that base layer is downloaded and stored once, not three times. Reordering Dockerfile instructions so rarely-changing steps (installing packages) come before frequently-changing ones (copying your own config) keeps rebuilds fast.

3. Names, tags, and digests

An image reference has up to three parts:

[registry/]repository[:tag]
myapp:1.0
docker.io/library/debian:bullseye-slim
PartMeaning
repositoryThe image's name, e.g. myapp
tagA human label for a specific version, e.g. 1.0 or 11.4. Defaults to latest if omitted.
digestA content hash (sha256:...) — the only fully unambiguous way to pin an exact image, since tags can be overwritten.
Pitfall: latest is just a tag like any other, not "the newest version" — it's whatever the maintainer happened to push as latest. Pinning a real version (or a digest) instead of floating on latest is what makes a build reproducible months later.

4. Registries and pulling

A registry is a server that stores images (Docker Hub is the default public one). docker pull downloads an image's layers from a registry; docker push uploads them. Layers you already have locally aren't re-downloaded — Docker checks layer hashes first. 

docker pull debian:bullseye-slim
docker pull alpine:3.20
Minimal base images like debian or alpine are the normal starting point for a custom build. The distinction that matters is between pulling a thin OS base layer you build on top of, versus pulling a full pre-packaged application image and using it as-is.

5. Building your own (preview of Tutorial 6)

docker build reads a Dockerfile and executes its instructions one by one, committing a new layer after each one, to produce a new image. Tutorial 6 covers Dockerfile syntax in depth — for now, just see the shape of the command: 

docker build -t myapp:1.0 ./myapp

6. Inspecting images

# List local images
docker images

# See every layer and the command that created it
docker history myapp:1.0

# Full metadata: env vars, entrypoint, exposed ports, layer digests
docker inspect myapp:1.0

7. One service, one image

In a multi-service Compose project, each service directory builds to its own tagged image — typically <service>:<tag> via docker-compose.yml's build: key pointing at that service's Dockerfile. That's why docker images after a successful build shows one image per service instead of one giant shared image.

One image, many containers: if you scale or restart a service, Compose reuses the already-built image and just creates a fresh container from it — it doesn't rebuild unless the Dockerfile or build context changed.

8. Useful commands

# List, remove, and clean up images
docker images
docker rmi myapp:1.0
docker image prune          # remove dangling (untagged) images
docker image prune -a       # remove all unused images

# Tag an existing image under a new name
docker tag myapp:1.0 myapp:backup

# Save/load an image as a tarball (no registry needed)
docker save -o myapp.tar myapp:1.0
docker load -i myapp.tar
