1. Why env vars aren't enough for passwords

An environment variable set on a container is readable by docker inspect, shows up in /proc/<pid>/environ inside the container, and is inherited by every child process the entrypoint spawns — including ones that might log their full environment on crash. None of that is acceptable for a database root password or a TLS private key. Docker secrets exist specifically to give sensitive values a narrower, file-based exposure instead.

2. What a Docker secret is

A secret is a piece of data (usually file contents) that Docker mounts as a read-only file at /run/secrets/<name> inside only the containers that explicitly request it — never as an environment variable, never baked into an image layer. In plain docker compose (no Swarm), this is implemented as a bind mount of the source file into that path; it's simpler than Swarm's encrypted-at-rest secret store, but it still achieves the core goal of keeping the value out of environment:/image layers.

3. Declaring and mounting secrets in Compose

Two parts, same pattern as networks and volumes: declare once at the top level, attach per-service.

secrets:
  db_password:
    file: ./secrets/db_password.txt
  db_root_password:
    file: ./secrets/db_root_password.txt

services:
  mariadb:
    secrets:
      - db_password
      - db_root_password

Inside the mariadb container, that produces:

/run/secrets/db_password        # contains exactly the contents of db_password.txt
/run/secrets/db_root_password

4. Reading a secret inside the container

An entrypoint script (or an app that supports it natively) just reads the file: 

#!/bin/sh
DB_PASSWORD=$(cat /run/secrets/db_password)
# ... use $DB_PASSWORD to set up the MariaDB user, then unset it from the shell ...
Many official images support a *_FILE convention (e.g. MYSQL_PASSWORD_FILE=/run/secrets/db_password) so you don't even need custom shell logic — the app reads the secret file itself at startup.

5. A real secrets block, explained

A secrets/ directory holds the actual sensitive files, declared once in docker-compose.yml:

secrets:
  db_password:
    file: ./secrets/db_password.txt
  db_root_password:
    file: ./secrets/db_root_password.txt
  nginx_crt:
    file: ./secrets/nginx.crt
  nginx_key:
    file: ./secrets/nginx.key
  wp_user_password:
    file: ./secrets/wp_user_password.txt
  wp_admin_password:
    file: ./secrets/wp_admin_password.txt

and attached only to the services that need each one:

  • mariadb gets db_password / db_root_password — never nginx_key.
  • nginx gets nginx_crt / nginx_key for TLS termination — never any database password.
  • wordpress gets the WP user/admin passwords — not the TLS key or the database root password.
This is least-privilege in practice: each container's /run/secrets/ contains only what that specific service needs. Compromising the WordPress container doesn't hand over the database root password, because it was never mounted there in the first place.

6. TLS termination at nginx

"TLS termination" means the encrypted connection ends at nginx — it decrypts the traffic and proxies plain HTTP to the services behind it (wordpress, etc.) over the internal network. The certificate and key that make this possible are exactly the kind of sensitive file-based data secrets exist for, which is why nginx_crt/nginx_key showed up in section 5. 

# Self-signed cert/key, good enough for local dev or an internal-only service
openssl req -x509 -nodes -days 365 \
  -newkey rsa:2048 \
  -keyout ./secrets/nginx.key \
  -out ./secrets/nginx.crt \
  -subj "/CN=localhost"

Inside the nginx image, the server block references the mounted secret paths directly:

server {
    listen 443 ssl;
    ssl_certificate     /run/secrets/nginx_crt;
    ssl_certificate_key /run/secrets/nginx_key;
    ssl_protocols       TLSv1.2 TLSv1.3;
    ...
}
Why pin ssl_protocols to TLS 1.2/1.3: nginx will otherwise still accept older protocol versions (TLS 1.0/1.1, SSLv3) that have known, practical cryptographic weaknesses. Explicitly listing only 1.2 and 1.3 isn't a formality — without it, an outdated client (or an attacker doing a downgrade attempt) can still negotiate a connection over a protocol version that's broken.
A self-signed cert is fine for this exercise — browsers will warn because no public CA vouches for it, but the encryption itself is identical to a CA-signed cert. The browser warning is about trust/identity verification, not about the strength of the TLS session.

7. Secret hygiene that actually matters

  • Never commit srcs/secrets/*.txt to git — they should be in .gitignore, generated locally or by a setup script, not checked into version control.
  • Never pass secrets via ARG/ENV in a Dockerfile — both end up baked into the image layers and visible to anyone with the image, permanently.
  • File permissions matter — your nginx.key being 600 (vs. the other files' 644) is intentional: a private key shouldn't be world-readable on the host even before Docker touches it.
  • Don't echo secrets in entrypoint scripts — a stray echo $DB_PASSWORD or set -x with a secret in scope will land the value in docker logs, defeating the entire point.

8. Useful commands

# Confirm what's actually mounted in a container's /run/secrets
docker exec mariadb ls -la /run/secrets

# Confirm a secret is NOT leaking via env or image layers
docker exec mariadb env | grep -i pass     # should show nothing
docker history mariadb:1.0                 # should show no password strings

# Validate the resolved compose config references the right files
docker compose config | grep -A2 "secrets:"
