1. Why environment variables, not hardcoded config

The same image (e.g. wordpress:1.0) might need a different DB host, port, or domain depending on where it's deployed. Baking those values into the image means rebuilding it for every change. Environment variables let you inject configuration at container-start time, keeping the image generic and reusable while the deployment-specific values live outside it.

2. Where env vars can come from

SourceMechanism
DockerfileENV KEY=value — baked into the image, becomes the default for every container from it
docker run-e KEY=value on the command line
Compose environment:Inline list/map in docker-compose.yml
Compose env_file:Points at a file (e.g. .env) of KEY=value lines, loaded into the container
Shell environmentCompose substitutes ${VAR} in the YAML itself from your shell or a .env file next to the compose file

3. Precedence when sources overlap

From lowest to highest priority: Dockerfile ENV < Compose env_file: < Compose environment: < a value passed with docker run -e/docker compose run -e. The more specific, more runtime-adjacent source always wins.

4. Variable substitution inside docker-compose.yml

Compose treats the YAML file itself as a template: before parsing, it substitutes ${VAR} placeholders using variables from your shell environment or from a .env file sitting next to docker-compose.yml. This is a different mechanism from env_file:, which only injects variables into the container — substitution happens earlier, in the YAML text itself. 

ports:
  - "${NGINX_PORT}:${NGINX_PORT}"
This is exactly how a single .env file can both configure the container's runtime environment (via env_file:) and drive structural parts of the compose file itself, like which port gets published — one source of truth, two different consumers.

5. A real .env, explained

A typical project-root .env holds keys like:

NGINX_PORT=...
MARIADB_PORT=...
WP_DB_USER=...
DB_HOST=...
DB_NAME=...
REDIS_PORT=...
FTP_PORT=...
ADMINER_PORT=...

Each service block references .env via env_file: - .env, and the compose file itself uses ${NGINX_PORT}-style substitution for port mappings. Splitting config this way means:

  • Changing a port or hostname is a one-line edit in .env, no Dockerfile rebuild needed.
  • Every service that needs DB_HOST=mariadb reads the exact same value — no risk of one service's config drifting from another's.
Notice what's missing from .env: there are no *_PASSWORD keys in the list above. Passwords belong in Docker secrets instead — see Tutorial 9 for why that split exists and isn't just stylistic.

6. Env vars vs. secrets — why split them

Environment variables are visible to anyone who can run docker inspect on the container, and they often leak into logs, crash dumps, or child-process environments. They're fine for non-sensitive config (ports, hostnames, usernames) but a poor fit for passwords and keys — which is precisely the gap Docker secrets (Tutorial 9) fill.

7. Useful commands

# See the fully-resolved compose config, with all ${VAR} substitution applied
docker compose config

# Check what env vars actually landed inside a running container
docker exec wordpress env

# Inspect just the env vars baked into an image (from its Dockerfile)
docker inspect -f '{{.Config.Env}}' wordpress:1.0
